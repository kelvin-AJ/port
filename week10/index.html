Notes from reading exercises and links to practice.

  • Form Validation
    1. Validation done on the browser is called client-side validation while validation done in the server is called server-side validation.
    2. We validate data to:
      • Get the right data in the right format
      • To protect users data such as ensuring a particular password format
      • To protect ourselves from malicious internet users
    3. We can validate on the client side using Built-in HTMl form validation(which has better performance than javascript) or using javascript(which is completely customizable).
    4. Form data that pass validation can be acces with the :valid CSS pseudo-class.
    5. Example:
      I don't like you
    6. The contraint validation API gives us enormous control over the user interface while validating form data with javascript.
    7. We can validate form data without using special APIs by writing our own javascript.
  • Using the Fetch API
    1. 'The Fetch API provides a JavaScript interface for accessing and manipulating parts of the protocol, such as requests and responses.'
    2. we can use the fetch() API with one argument, the url we are fetching from. or passing in request options as a second argument.
    3. the options object has the following fields: method('GET','POST','PUT','DELETE'), mode, cache, credentials, headers, redirect, body(which must match the "Content-Type" header. If Content-Type is set 'application/json', we can pass in a JSON string has the body)
    4. We can also upload files with the fetch API using FormData()
    5. We can supply our own request object with our own headers to make fetch requests. Header contenents can be queried and retrieved.
    6. 'A good use case for headers is checking whether the content type is correct before you process it further.'
    7. The most common response properties used are :
      • Response.status which contains the response status code. (usually, 200- 299 is good, 400- is bad)
      • Response.statusText which corresponds to the HTTP status code message.
      • Response.ok returns a boolean for whether a response is okay or not.
    8. Rensponses can also be created via JavaScript, but are only really useful in serviceWorkers(proxy servers between web applications), when providing a custom response using the respondsWith() method.

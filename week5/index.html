  • Testing and Debugging
    1. Errors are usually caused by: the system or external devices that interact with the program, problems in the program such as wrong syntax or faulty logic, or incorrect data entered by the user that the program cannot handle.
    2. exceptions are errors that produce return values that we can use to handle those errors.
    3. A stack trace is s sequence of method calls that lead to the point where the error occurred.
    4. An expert programmer should let errors occur loudly during development so they can be identified and fixed quickly. While in production, errors should occur gracefully, so the user experience is not affected.
    5. Writing code in strict mode can help to improve its clarity and speed since it follows conventions and will throw exceptions if any sloppy code practices are used.
    6. We activate strict mode by adding the string 'use strict' at the first line of our script file
    7. We can also use strict mode for individual functions by adding 'use strict' as the first line of the function.
    8. Linting tools test the quality of code beyond strict mode. They are useful in enforcing a programming style guide, especially when working with a team so everyone follows the same conventions.
    9. To implement feature detection, we wrap whatever 'dangerous' method we want to use in if statements that check if the object exists before running them.
    10. When debugging, we run through the program to see what happens at different points of its execution. With breakpoints, we can pause the execution at specific points and view the values of different variables at that point in the program's execution.
    11. we can use alerts to set breakpoints because the program actually stops running until the ok button is clicked.
    12. We can also use the console by logging the value of variables at strategic points in our program. This does not stop code execution like the alert.
    13. debugger command sets breakpoints in the program so the debugger pauses execution at those points.
    14. error objects can be created by the host or in the code when an exception occurs by using the error constructor function:
      const error = new Error('error message goes here');
    15. The 'throw' statement causes the program to stop
    16. If we suspect a piece of code will result in an exception, we can wrap it in a try block. This will run the code inside the block as normal, but if an exception occurs it will pass the error object that is thrown onto a catch block.
    17. A finally block can be added after a catch block, anything in the finally block would be executed regardless of whether an extension occurred or not.
    18. Test-driven development (TDD) is the process of writing tests before any actual code. Following this principle results in the following workflow:
    19. Write tests (that initially fail), Write code to pass the tests, Refactor the code, Test refactored code, and Write more tests for new features.
